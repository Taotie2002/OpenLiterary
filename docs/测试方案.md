
### 完整的流程测试方案 (End-to-End Test Plan)

鉴于该系统将在具有 16GB 统一内存约束的 Apple Silicon 架构上运行本地 MLX 推理引擎，测试方案必须兼顾**状态机容错**与**硬件边界压榨**。

建议按照以下四个阶段（Phases）进行递进式验证：

#### Phase 1: 冷启动与连通性验证 (Smoke Testing)

**目标**：验证数据库建表、文件 I/O 切分以及 Mock 引擎的流水线闭环。

1. **配置准备**：确保 `SYS_CONFIG["llm_backend"] = "mock"`。
2. **数据准备**：在项目根目录创建 `input/ch01.md`，填入一段 3000 字左右的测试文本。
3. **初始化拦截测试**：
```bash
python3 translator_agent.py init --chapter ch01

```


*预期：终端打印成功切分的 chunk，并成功在 `db/` 下生成 `.db` 和 `.sqlite` 文件。*
4. **全链路空跑测试**：
```bash
python3 translator_agent.py pipeline --chapter ch01

```


*预期：系统在 5 秒内极速跑完全程。观察 `output/ch01/` 目录下是否按预期生成了 `_raw.json`, `_literary.json` 和 `_final.json`，且包含 Mock 数据。*

#### Phase 2: 状态机防弹测试 (Chaos Engineering)

**目标**：验证在极不稳定环境下，系统能否准确记录断点并自动重试/回溯。

1. **强制中断恢复**：
* 将后端切换为 `openai_api`，并故意填错 `openai_api_base` 端口。
* 运行 `python3 translator_agent.py pipeline`。
* *预期：系统在各阶段抛出连接异常，重试 3 次后将 chunk 标记为 `PERMANENTLY_FAILED`。*


2. **强制复写覆盖测试**：
* 修改 `input/ch01.md` 原文。
* 执行 `python3 translator_agent.py init --chapter ch01 --force`。
* *预期：终端提示 `🔄 --force 模式：覆盖更新 X 个已有 chunk`，并将这些 chunk 的状态重置为 `PENDING`，重试次数清零。*


3. **决策树回溯测试**：
* 在系统完成翻译后，手动用 SQLite 工具向 `decision_db` 插入一条具有冲突的规则（关联特定的 chunk_id）。
* 再次运行 Pipeline。
* *预期：系统识别到 DIRTY 标记，自动将受影响的 chunk 退回 `EXTRACTING_TERMS` 阶段重跑，同时清除旧的 output 文件。*



#### Phase 3: 硬件边界极限压测 (Hardware Stress Testing)

**目标**：试探 MLX 在统一内存架构下的驻留极限，验证自动卸载机制是否能防止 OOM (Out of Memory) 崩溃。

1. **配置环境**：确保 `SYS_CONFIG["llm_backend"] = "mlx"`，并下载好量化模型（如 `Qwen2.5-7B-Instruct-MLX-4bit`）。
2. **执行专项内存脚本**：
```bash
python3 translator_agent.py memory

```


3. **观察指标**：
* **测试 3 (内存压力模拟)**：系统能否在检测到内存压力时，成功触发 `auto_unload_if_needed()`，并在日志中打出 `🧹 [MLX] 正在从统一内存中卸载模型...`。
* **测试 4 (连续吞吐)**：观察平均 `tok/s` 是否在合理区间，并监控 Mac 的“活动监视器”，确认内存图谱呈规律性锯齿状（释放与加载），而非单调递增导致红线。



#### Phase 4: 文学指标回归测试 (Golden Set Regression)

**目标**：验证 Agent 提示词链条的实际翻译品味，量化风格坍缩率。

1. **构建黄金集**：在 `input/golden/hyperion_5k.md` 中放入原著中包含长难句、密集典故（如 Keats）和极强宗教隐喻的高难度文本片段。
2. **执行评测**：
```bash
python3 translator_agent.py golden

```


3. **基线确认**：
* 打开 `output/golden_test_report.json`。
* 重点核对 `avg_style_collapse_rate`（建议底线：低于 40% 即可视为可用）和 `pass_rate`（Judge Agent 的一次性通过率）。这决定了系统在生产环境中的人工干预成本。
